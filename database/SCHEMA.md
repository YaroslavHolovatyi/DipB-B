# Beer & Beverages ‚Äî Database Schema

PostgreSQL 16 schema for the mobile rebuild (React Native + Expo / FastAPI / PostgreSQL).
Companion to `schema.sql`.

## Why PostgreSQL (vs. the original MySQL 8)

The data model is heavily relational (users ‚Üî friends, friend groups ‚Üî members, quiz answers ‚Üî races with scores, receipt items ‚Üî participants). Postgres keeps that intact while adding three specific wins for this app:

- **PostGIS** ‚Äî `GEOGRAPHY(POINT, 4326)` on `cities`, `bars`, and `raids` lets us run "bars/raids near me" queries with a real spatial index. Useful for `expo-location` on the phone.
- **JSONB columns** ‚Äî receipt OCR responses, achievement requirements, push-notification payloads, and quiz score breakdowns are stored as JSON without needing extra tables.
- **Full-text search** ‚Äî bars have a maintained `tsvector` column + GIN index for fast catalog search across name, description, and address; a `pg_trgm` index handles fuzzy "name contains" queries.

## Conventions used everywhere

- `BIGINT GENERATED BY DEFAULT AS IDENTITY` for primary keys (Spring/JPA `IDENTITY` equivalent).
- `TIMESTAMPTZ` for every timestamp; server runs in UTC, clients convert.
- `citext` for `users.email` and `users.username` so uniqueness is case-insensitive.
- `created_at` / `updated_at` on mutable rows, with a shared `set_updated_at()` trigger.
- `deleted_at TIMESTAMPTZ` (soft delete) on user-facing entities: `users`, `bars`, `checks`.
- Junction tables use **composite primary keys** (e.g. `(user_id, bar_id)` on `bar_favorites`).
- All foreign keys declare `ON DELETE` policy explicitly (`CASCADE` / `RESTRICT` / `SET NULL`).
- Enums (`CREATE TYPE ‚Ä¶ AS ENUM`) for closed value sets ‚Äî `user_role`, `price_category`, `friendship_status`, `raid_status`, etc.

## Table groups

### 1. Geography & reference data
`cities`, `vibes`, `drinks`, `races`, `race_drinks`, `race_vibes`.
Static lookup tables seeded once. `cities.location` is PostGIS, so we can later add `WHERE ST_DWithin(...)` queries.

### 2. Users & auth
`users` (1 row per account), `refresh_tokens` (one row per active JWT refresh ‚Äî stored as a SHA-256 hash, never raw), `push_tokens` (Expo / APNs / FCM device tokens for push notifications). `users.race_id` is nullable: it's only set once the user completes the quiz.

### 3. Bars
`bars` is the main catalog. It maintains a `search_vector` column updated by a trigger so search queries hit a GIN index rather than scanning the table. `rating_avg` and `rating_count` are denormalized caches refreshed when a review is created/updated (a follow-up trigger can be added, or the FastAPI layer can update them ‚Äî TBD when we wire up reviews).
Companions: `bar_vibes` (many-to-many with `vibes`), `bar_photos`, `bar_reviews` (one review per user per bar, enforced by `UNIQUE (bar_id, user_id)`), `bar_favorites` (composite-PK join).

### 4. Quiz
`quiz_questions` (ordered via `position`), `quiz_answers` (ordered inside a question), `quiz_answer_races` (each answer awards points to one or more races ‚Äî `(answer_id, race_id, score)`), `user_quiz_results` (each quiz run is recorded, so users can retake; `score_breakdown` JSONB stores the per-race totals for auditability).

### 5. Social
- `user_friends`: symmetric friendship. On accept, we insert two rows so "find my friends" is one indexed read. `CHECK (user_id <> friend_id)` blocks self-friending.
- `friend_requests`: a partial unique index (`WHERE status = 'pending'`) prevents duplicate pending requests between the same pair, while still allowing past declined/cancelled ones to exist.
- `friend_groups` + `friend_group_members` ("Party for Dungeon"). `owner_id` for control, `role` enum (`member` / `admin`) inside the group.

### 6. Raids (events / planned gatherings)
`raids` represent a planned meet-up at a bar (or at a freeform location). `bar_id` is nullable for off-catalog spots; a PostGIS `location` lets the "Raids near you" feed work without a bar reference. RSVP via `raid_participants` with `status` enum (`going` / `maybe` / `declined`). Partial index `WHERE status IN ('planned','ongoing')` keeps the upcoming-raids feed query fast.

### 7. Checks (receipts) and the collaborative split room
- `checks` ‚Äî header row per uploaded receipt. `bar_id` nullable + `bar_name_freeform` for receipts from places not in our catalog. `ocr_payload JSONB` stores the raw Vision API response for audit / re-parse.
- `check_items` ‚Äî parsed line items.
- `check_participants` ‚Äî people in the split. `user_id` is nullable so guests with no account can still be assigned items. New `status` enum (`invited` ‚Üí `joined` ‚Üí `ready` ‚Üí `left`) drives the split-room flow: registered users start as `invited` and a `check_invite` notification fires; guests are inserted as `joined` directly because they don't have an account. Timestamps `joined_at`, `ready_at`, `left_at` log each transition. The room is considered closed when every active participant is `ready` and all items are fully assigned.
- `check_item_assignments` ‚Äî composite-PK join `(check_item_id, participant_id)` with `quantity` and `amount`; this is what powers the "‚Ç¥X owed by Y" output. Updated incrementally over the WebSocket as participants claim items live.
- `dice_proposals` ‚Äî opens the unanimous-consent vote for the D20 "Choose One to Pay for All" game. `status` is `pending` ‚Üí `accepted` (every vote = `accept`, dice will roll) or `declined` (any vote = `decline`, game off) or `cancelled` (participant left, proposer cancelled) or `completed` (dice rolled and a winner recorded). A partial unique index `(check_id) WHERE status='pending'` ensures only one proposal exists per check at a time.
- `dice_proposal_votes` ‚Äî composite-PK join `(proposal_id, user_id)` with a `vote` enum (`pending` / `accept` / `decline`). One row per registered participant; guests don't vote.
- `kind_soul_events` ‚Äî captures the dice outcome. One per check (UNIQUE). Linked back to its `dice_proposals` row via `proposal_id` for the audit trail. Drives the "–î–æ–±—Ä–∞ –î—É—à–∞" leaderboard.

### 8. Achievements
`achievements` defines what can be earned (`code` is the stable string ID used in code, e.g. `KIND_SOUL_10`). `requirement JSONB` lets us encode rules like `{"type":"check_count","threshold":10}` without a schema change per achievement type. `user_achievements` is the composite-PK join with an optional `progress JSONB` so we can show "7/10 receipts uploaded" before unlock.

### 9. Notifications
One table covers in-app + push. `type` enum routes display logic; `data JSONB` carries everything else (raid id, friend id, etc.). `related_entity_type` + `related_entity_id` give a generic deep-link target. Partial index on unread rows keeps the badge query cheap.

### 10. Chat & messaging (real-time, with raid chats and message-state ticks)
- `conversations` ‚Äî one row per direct (1-on-1) or group conversation. Three optional FK links classify the chat: `friend_group_id` (UNIQUE) ties a group chat to a saved friend group so membership stays in sync; `raid_id` (UNIQUE) ties a group chat to a raid so the chat exists for every raid automatically and follows raid RSVPs; both null means an ad-hoc group chat or a direct conversation. `last_message_at` is bumped by a trigger when any message is inserted, so chat-list sort is O(1) read.
- `conversation_participants` ‚Äî composite PK `(conversation_id, user_id)`. Two watermark columns drive the message-state ticks: `last_delivered_at` (newest message confirmed to have reached this user's device ‚Äî bumped on a `delivery.ack` WebSocket frame or a successful Expo Push receipt) and `last_read_at` (newest message this user has read ‚Äî bumped when the user opens the conversation). `left_at` lets a user leave a group without losing history. Partial index on `WHERE left_at IS NULL` keeps the "my active conversations" query fast.
- `messages` ‚Äî body and/or attachments (JSONB array of `{url, type, name, size}`). `reply_to_id` self-references for threaded replies. `deleted_at` for soft delete. Composite index `(conversation_id, created_at DESC)` for paginated message lists. `CHECK` constraint enforces non-empty content. A message's tick state (sent / delivered / read) is computed at query time by comparing `messages.created_at` against the other participants' `last_delivered_at` / `last_read_at`.
- `message_reactions` ‚Äî composite PK `(message_id, user_id, emoji)` so a user can react with multiple emojis.
- `user_presence` ‚Äî one row per user, updated by the WebSocket layer (`status` enum and `last_seen_at`). Used for the green-dot indicator in the chat list.

Chat membership for raid chats is kept in sync with raid RSVPs at the application layer (or via a deferred trigger): inserts on `raid_participants` with `status IN ('going','maybe')` insert a matching `conversation_participants` row; deletions or transitions to `declined` set `left_at`.

Chat messages do **not** create rows in `notifications` (that would create heavy churn). Instead, the WebSocket layer pushes the message directly to online recipients; offline recipients get a single Expo Push notification whose successful receipt also drives the `last_delivered_at` update.

### 11. Tavern Tales ‚Äî AI Dungeon Master
- `dnd_class_info` ‚Äî reference table seeded with the twelve 5e player classes. `slug` is of type `dnd_class` (enum) and acts as the primary key; `hit_die` is constrained to {6, 8, 10, 12}. Surfaced by `GET /api/dnd/classes` to drive the character-creation picker.
- `dnd_characters` ‚Äî one row per character. `user_id` and `race_id` link back to the existing tables (so the race comes for free from the quiz result on `users.race_id`). `class_slug` enforces FK to `dnd_class_info`. Stats live in a JSONB blob (`{"str":14,...}`) so we don't have six columns to migrate when 5e adds an option. HP, AC, level, XP, alignment, inventory, and known spells round out the sheet. `last_played_at` indexed (descending, nulls last, partial on not-deleted) so the "your characters" list is a cheap read.
- `dnd_sessions` ‚Äî one row per quest. `mode` is the three-mode enum that controls the AI's system prompt. `summary` is a rolling text summary the AI maintains so older turns can be evicted from the live context window once the message log exceeds a configurable threshold. `metadata JSONB` holds world state (location, objective, NPCs met). `input_tokens_used` / `output_tokens_used` track per-session cost. A **partial unique index** `(character_id) WHERE status IN ('active','paused')` enforces one open quest per character at a time.
- `dnd_messages` ‚Äî the turn log. `role` distinguishes `user`, `assistant`, `system`, `dice_roll`, and `narration` so the UI styles them differently. `metadata JSONB` holds structured payloads ‚Äî for `dice_roll`: `{"dice":"d20","modifier":3,"result":17,"purpose":"persuasion"}`; for `assistant` turns the AI optionally embeds `state_change` (HP delta, loot diff) or `request_roll` blocks the client renders as a "Roll" button. `tokens_in` / `tokens_out` per row.
- `dnd_usage_quota` ‚Äî one row per user (PK is `user_id`). Daily and monthly token caps with reset dates. The API rejects new turns with a `429` and the quota payload when a cap is hit; a cron resets the counters at the boundary.

Tavern Tales is single-player only in MVP. The schema deliberately does not introduce party / co-op tables; a future "party play" extension would add a `dnd_party` join layer without changing any existing column.

## Indexing strategy summary

| Index type | Where | Why |
|---|---|---|
| GiST | `cities.location`, `bars.location`, `raids.location` | spatial "nearby" queries |
| GIN | `bars.search_vector` | full-text bar search |
| GIN trigram | `bars.name` | fuzzy name match |
| Partial | `users(is_active) WHERE deleted_at IS NULL` | active-user lists |
| Partial | `friend_requests(sender_id, recipient_id) WHERE status='pending'` | dedupe inbound requests |
| Partial | `notifications(recipient_id) WHERE read_at IS NULL` | unread badge |
| Partial | `raids(scheduled_at) WHERE status IN ('planned','ongoing')` | upcoming feed |
| Composite | `idx_checks_user (user_id, created_at DESC)` | profile receipt history |
